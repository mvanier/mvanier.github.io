This document will explain how to set up and run a Docker image with the course software installed.

Docker is super cool, but our experience with it is limited. In addition, running Docker successfully requires more sophistication than the other approaches we’ve described for setting up the course software.

We recommend that, for now, you only try to set up Docker if all the other options have failed.

In the future, though, this will probably become the standard way to set up software for CS 11 (and other courses) since it’s extremely portable between operating systems.

What Docker is

Docker is a software suite that runs software in containers.

A container is basically an ultra stripped-down virtual machine. A virtual machine is an operating system running inside another operating system, like Linux running inside Windows or Mac OS.

Most people who have used VMs are familiar with them from something like VirtualBox VMs, which typically run a full version of some kind of Linux (like Ubuntu Linux) inside of Windows or Mac OS. But most of the time, the vast majority of software you have in a VM is software you don’t need. What if you could only have the bare minimum of software packages to run some application? Then your VM would be smaller and would start up faster. That’s the idea of containers. (There are other advantages as well, but they don’t concern us.)

In our case, we don’t actually want to run a single program; we want to run a small number of programs related to compiling C code. But the idea of containers can work just as well in this case. Docker was originally designed as a way to "deploy" specific applications (often web applications) but it’s becoming more popular as a way of developing programs, which is how we’ll use it.

Containers stored in your computer are generally referred to as images, and we’ll use that term below.

Setting up Docker

Go to the Docker web site. Follow the links to the Get Started page. Install Docker Desktop.

When Docker Desktop is installed, you will be able to see the Docker "dashboard". This will show you all the containers that are running on your computer, and allow you to manipulate them in various ways.

In addition to that, Docker will also install a command-line tool called docker; we will be using this exclusively.

Since the primary use case for Docker is currently Macs with M1 chips, we’re going to assume a Mac terminal environment. You can use the same commands on Windows, but there may be small differences, and if you are using Windows, you don’t need Docker for this course anyway.

Dockerfiles

To create a container in Docker, you need what is called a Dockerfile. This is a file (with the exact name Dockerfile) that contains information on how to set up a particular container. We will supply the Dockerfile for this course, but we don’t expect you to understand what’s in it.

In our case, we want to install a container with a minimal Ubuntu Linux image that also contains the gcc C compiler, the make program and the gdb debugger.

Here is the Dockerfile (also downloadable here):

FROM        ubuntu:hirsute
MAINTAINER  mvanier

# update and install dependencies
RUN         apt-get update \
                && apt-get install -y \
                    software-properties-common \
                    wget \
                && apt-get update \
                && apt-get install -y make nano vim gcc gdb

WORKDIR     /cs11_c

(Please download this file instead of copying-and-pasting!)

What this will do, when installed, is to create an Ubuntu Linux VM that also contains the programs make, gcc, gdb, nano, and vim. What do these programs do?

  • gcc is the C compiler. This is the most important program we’re going to install.

  • make is a program to make it easier to compile code. We’ll be using this starting in lab 2.

  • gdb is the C debugger. We won’t use this until lab 6.

  • nano and vim are source code editors. You don’t actually have to use them, but you can use them to edit source code from right inside the container. They only work inside the terminal. (There are versions of vim that work in a separate window, but not this version.) vim is great, but complicated. nano is dirt simple.

Building the image

Once you have the Dockerfile downloaded, make a directory for your CS 11 code and move the Dockerfile into that directory. We will refer to this directory as your "Docker directory".

Start up a terminal and navigate to your Docker directory. Then run this command:

$ docker build -t cs11_c .

If you’ve set up Docker correctly, this will download some software, print out some stuff, and create your image. This should take no more than a minute or two. Ideally, you will never have to do this again.

Running the image

There are many different ways to run a Docker container from an image. We recommend typing the following command:

$ docker run --rm -it --name=cs11_c -h caltech-cs11 -v $PWD/files:/cs11_c cs11_c bash

Let’s walk through this.

The --name=cs11_c part identifies the image that you are going to run. In this case, it’s the cs11_c image you just created.

The -h caltech-cs11 part gives a "hostname" to the running container (the name of the VM); it’s caltech-cs11.

The -v $PWD/files:/cs11_c part will create a subdirectory called files (or use it if it’s already there) and in the VM it will be referred to as the /cs11_c directory. This is where you can edit your files (even from outside of the container!).

The bash part starts up the bash shell, so you have a Linux command-line environment.

Once this runs, you will see a prompt like this:

root@caltech-cs11:/cs11_c#

You are now in a Linux container and can compile C code and run it!

Stopping the image

You can get out of the image by simply typing Control-D. Note that when you want to restart the image, you’ll have to re-type the entire command described above. (You can also write a shell script to do this or create a shell alias.)

Removing the image

If you want to remove the image for some reason, you can type

$ docker image rm cs11_c

Don’t do this unless there is a really good reason to do it. If you do, you’ll have to rebuild the image to use it again.

Starting over

If you’ve really messed things up and want to start over from scratch, type this:

$ docker system prune -a

This will remove all images and everything else stored inside Docker. (You won’t have to download Docker again, though.)

Editing and running your code

Now comes the fun part: writing code! If you’ve set things up as we’ve described above, it’s easy to edit code outside of the Docker container and run it from inside the container. Why would you want to do that? Mostly because you want to run a nice editor (like Visual Studio Code) to write your code, and this isn’t available inside the container.

In your Docker directory, there should be a subdirectory called files. That’s where your C source code files will go. You can make subdirectories there, like lab1, lab2, etc. and when you start the Docker container, it will be able to see them. Save your files to these directories. Then, inside the Docker container, you can compile them.

For instance, if you’ve saved a file in your Docker directory as files/lab1/hello1.c, inside the container you can compile it:

root@caltech-cs11:/cs11_c# cd lab1
root@caltech-cs11:/cs11_c# gcc -o hello1 hello.c
root@caltech-cs11:/cs11_c# ./hello1
hello, world!

And there you are! Docker is awesome!
