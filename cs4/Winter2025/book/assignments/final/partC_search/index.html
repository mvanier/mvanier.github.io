    Part C: The Search module (30 points)

    The code for this section goes in the file search.ml.

    If everything has gone well so far, you can run the interactive Knight's tour solver, but you can't yet get the computer to generate solutions for you. In this section you will write the code to do that.

    Inside the search.ml template file there is this code:

    open Storage
open Board
open Utils

exception Solution_not_found

module type Searcher =
  sig
    val search : int -> int -> int -> int -> bool -> (Loc.t list) option
  end

module Make (S : Storage) : Searcher =
  struct
    module B = Board.Make(S)
    module P = Print(B)

    (* Helper functions go here. *)

    let search nrows ncols start_row start_col print =
      failwith "TODO"
  end

    The Make functor takes a Storage implementation as its argument. For instance, if you wanted to use ImpStorage, you would write:

    module SI = Search.Make(ImpStorage)

    This is exactly what is done in knights_tour.ml.

    Inside the Make functor, the S module stands for the particular implementation of Storage that is being used. Then a B module is created. This is an implementation of the Board module type, using the Board.Make functor parameterized on the S module. Finally, a P module is created from the Print functor, which is defined in the file utils.ml. This module contains a print function which allows us to print a readable representation of the board.

    Function to write

    There is only one function inside the Make functor that you need to write. Here it is.

    1. search

    [30 points]

    This function takes five arguments:

    • nrows, the number of rows in the board

    • ncols, the number of columns in the board

    • start_row, the row index of the first Knight to be placed

    • start_col, the column index of the first Knight to be placed

    • print, a boolean value which, if true, causes the final solved board to be printed at the end

    The function will do the following:

    • It will create a new empty board of dimensions nrows by ncols.

    • It will place a knight at the specified starting location (start_row, start_col).

    • It will attempt to find a Knight's tour on the board starting from that board configuration.

    • If a solution is found, it gets a list of all the placed locations in the order of their placement on the board (with the location of the first Knight placed at the head of the list, etc.), and returns it as an option value (i.e. Some <list>). If the print argument is true, it also prints out the solved board configuration.

    • If a solution is not found, it returns None.

    Here is an outline of the algorithm:

    1. Create the board and place the initial Knight.

    2. If the board is solved, stop and return the list of locations placed as an option value (Some <list>).

    3. Otherwise, start from the location of the last Knight that was placed on the board.

    4. Get all the available locations that can be reached from that location by a Knight's move, along with the reachability count of these locations (this is a (location, int) list). If there are no locations, the search fails, so return None.

    5. Compute the minimum reachability count of these locations, and collect only those locations that have that count.

    6. Pick one of those locations at random, and place a Knight there.

      Note

      You must pick the location at random to get full marks.

    7. Go back to step 2.

    One interesting feature of this algorithm is that there is no need for anything clever like backtracking! If a solution ever gets stuck, the algorithm fails immediately. In theory, you could solve this problem using backtracking, or you could combine backtracking with the Warnsdorff rule, and it would make the algorithm more likely to succeed. Despite this, this very naive algorithm will usually work fine as long as the board isn't extremely large. Our experiments show that when boards get large e.g. 100x100 and larger, the probability of getting a solution goes down considerably.

    Here are some hints and suggestions to help you write this function:

    • The Board functions from the B module will be an essential part of your solution.

    • You are expected to use the get_loc_counts_from_loc function in the Board implementation to find the reachability counts corresponding to particular locations.
      If you don't do that (e.g. if you just compute them on the fly), your function may still work, but you will lose half the points.

    • To pick a random element from a list, the Random.int and List.nth functions will be useful.

    • To print the board, use the P.print_board function.

    • You will probably find it convenient to write one or more helper functions for this section in addition to the search function. (This is not a requirement.)

    • Once you realize that a board is unsolvable, you can raise the Solution_not_found exception and catch it in another function that can return the None value. This can simplify the code.

    • Your code should only take a few seconds to solve a board, or else fail. If it takes more than (say) 30 seconds you are almost certainly doing something wrong.

      Note

      If your solver is extremely inefficient, you could lose up to half points. ("Extremely inefficient" means that running all the search tests takes more than 60 seconds. Our solution takes less than 1 second to run all the search tests.)

    • The List.fold_left function can significantly reduce the number of lines in your solution.

    Our solution is less than 40 lines of code.
