    Part C: Modules and functors (40 points)

    The purpose of this section is to give you practice working with modules and functors.

    There are two ways in which modules are typically used in OCaml.

    1. Each file of OCaml code defines a module, where the .mli file is the interface of the module and the .ml file is the implementation of the module. This is the way we have been using modules up until now in the course.

    2. A module can be defined inside of a .ml file; this usually represents some kind of data structure or unit of functionality. This use of modules is what we will be dealing with in this section.

    We will define a stack interface and two different implementations of stacks, one purely functional and one imperative. Then we will use stacks to do an interesting computation.

    Warning

    OCaml has a Stack module as part of its standard library. You are not allowed to use any part of it in this section.

    The STACK module type

    When working with modules, it's usually a good idea to define a "module type" which describes the capabilities of the module. The distinction between module and module type is exactly the distinction between .ml files and .mli files, except that modules and module types can occur inside .ml files.

    Here, we will be working with the following module type, which represents stacks:

    module type STACK =
  sig
    (* The type of stacks. *)
    type 'a t

    (* Create a new, empty stack. *)
    val make : unit -> 'a t

    (* Push a value to a stack, returning a new stack. *)
    val push : 'a -> 'a t -> 'a t

    (* Pop a value from a stack, returning either
       - the value popped, plus the remaining stack
       - None (to indicate that the stack was empty) *)
    val pop : 'a t -> ('a * 'a t) option
  end

    (You will need to copy this into your submission.)

    The code between the keywords sig (which stands for "signature") and end is the stack interface. It includes one type (the type of stacks), and three functions on stacks.

    The type of stacks is type 'a t because a stack is allowed to store any type of value ('a refers to the type of stack elements), as long as all the different values in the stack have the same type. The t is just a short name that refers to the stack type. (It's a convention in OCaml for modules that define a single datatype to use the letter t to refer to that datatype; we could have alternatively called it stack or anything else we wanted.)

    The functions on stacks have these meanings:

    1. make takes a unit value (()) and makes a new, empty stack.

    2. push takes a value and a stack and pushes that value onto the stack, returning a new stack.

    3. pop takes a stack and tries to pop the top value off the stack, returning the popped value and the remaining stack; if it succeeds, it returns Some (<value>, <stack>), and if it fails, it returns None.

      Writing the pop function this way means that we don't need another function to check if the stack is empty or not.

    The STACK module type doesn't actually allow us to create stacks; for that, we have to create modules that conform to that interface. In fact, we could create stack modules without defining a module type, but we will see below (when we get to functors) why having a module type is essential.

    1. Functional stacks

    [5 points]

    One of the simplest ways of implementing a stack in a purely functional way is as a list:

    1. To push something onto a list, put it at the front of the list using the :: list constructor, giving a new list.

    2. To pop something from a list, split it into its head and tail; the head is the value popped, and the tail is the remaining "stack". If the stack is empty, return None, and if not, return Some (<head>, <tail>).

    Here's a template for writing this as an OCaml module:

    module Fstack : STACK =
  struct
    type 'a t = 'a list

    let make () = failwith "TODO"

    let push v s = failwith "TODO"

    let pop s = failwith "TODO"
  end

    The line

    module Fstack : STACK = ...

    indicates that the name of the module is Fstack (for "functional stack") and it must conform to the STACK interface. You could define it like this:

    module Fstack = ...

    without the STACK module type; in that case, you could put whatever you like inside the module. Having the : STACK constraint does two things:

    1. It means that the module must implement all the types and functions specified in the STACK module type, with the correct types.

    2. It means that nothing other than the STACK types and functions are exported from the module.

    You can have extra functions in the module if you like, but nothing that uses the module will be able to access them. Similarly, the internal representation of the stack datatype will not be visible from outside the module.

    Here's what this would look like in the OCaml interpreter. Let's suppose we want to create a stack of ints, and add a couple of values to it.

    # let s = Fstack.make () ;;
val s : '_weak2 Fstack.t = <abstr>

# let s' = Fstack.push 1 s ;;
val s' : int Fstack.t = <abstr>

# let s'' = Fstack.push 2 s' ;;
val s'' : int Fstack.t = <abstr>

# Fstack.pop s'' ;;
- : (int * int Fstack.t) option = Some (2, <abstr>)

# Fstack.pop s' ;;
- : (int * int Fstack.t) option = Some (1, <abstr>)

# Fstack.pop s ;;
- : (int * int Fstack.t) option = None

    Notice that pushing a value to a stack returns a new stack, and popping a value off a stack also returns a new stack plus the popped value (unless the stack is empty).

    The one weird thing here is the type of the newly-created stack:

    # let s = Fstack.make () ;;
val s : '_weak2 Fstack.t = <abstr>

    Don't worry about this '_weak2 business. It just means that OCaml doesn't know enough at that point to know what the type of the stack elements are. The _weak2 name means "type to be determined later". If we wanted to specify the type up front, we could have written it with a type annotation:

    # let (s : int Fstack.t) = Fstack.make () ;;
val s : int Fstack.t = <abstr>

    The <abstr> value is also interesting. It's OCaml's way of saying "There is a value there, but it's abstract, which means that you can't know anything about what it is internally." The module type STACK prevents us from looking inside the stack to see anything about its implementation - this is called "data hiding" and helps enforce module boundaries. If we didn't use the STACK module type to constrain the module, you could see that the stack was implemented as a list, and it would have looked like this:

    # let (s : int Fstack.t) = Fstack.make () ;;
val s : int Fstack.t = []

    The implementation of the stack functions (make, push, and pop) is simple; make and push are trivial one-liners, while pop has to be a little longer to deal with the empty stack case.

    2. Imperative stacks

    [5 points]

    It's often useful to have multiple implementations of the same data structure. For this problem, define an imperative stack module. Here is the template:

    module Istack : STACK =
  struct
    type 'a t = 'a list ref

    let make () = failwith "TODO"

    let push v s = failwith "TODO"

    let pop s = failwith "TODO"
  end

    An imperative stack is implemented here as a reference cell storing a list. The type 'a list ref should be read as ('a list) ref i.e. a ref cell containing a value of type 'a list. Your job is to fill in the definitions of the make, push and pop functions. Again, make and push are very simple, while pop is a little bit more complicated.

    Note

    Most implementations of imperative stacks would redefine the types of push and pop to the following:

    val push : 'a -> 'a t -> unit

val pop : 'a t -> 'a option

    In other words, these functions would not return the updated stacks, since the stack updates would be reflected in changes to the contents of the ref cell.

    We aren't doing this here, because we want both the functional and imperative stacks to have the same interface. This will be useful when we discuss functors below. Nevertheless, the implementations of push and pop must change the ref cell contents (unless you are popping an empty stack, of course).

    A functor on stacks: compiling arithmetic expressions to postfix form

    Now that we have a stack module type and two different implementations of stacks, we can do something with them. We are going to do a simple computation to compile arithmetic expressions (the expr type we saw in assignment 4) to a lower-level form.

    Recall that the expr datatype looks like this:

    type expr =
  | Int of int           (* constant *)
  | Var of string        (* variable *)
  | Add of expr * expr   (* expr1 + expr2 *)
  | Mul of expr * expr   (* expr1 * expr2 *)
  | Pow of expr * int    (* expr^n *)

    Expressions of this type could be generated by parsing arithmetic expressions entered by a user in a calculator program, for instance. They could be interpreted directly, but it's not uncommon in actual programming language implementations to convert this representation to a "postfix" representation, where the expressions would become "instructions". Here is the target datatype:

    type inst =
  | Push_int of int      (* push an integer onto a stack *)
  | Push_var of string   (* push the value of a variable onto a stack *)
  | Op_add               (* add the top two values on the stack *)
  | Op_mul               (* multiply the top two values on the stack *)
  | Op_pow               (* take the power of the top two stack values *)

    These are called "postfix" instructions because (for instance) an input expression of the form

    Expr (Int 1, Int 2)

    would be converted to the list of expressions

    [Push_int 1, Push_int 2, Op_add]

    This puts the operator (+) after the operands (1 and 2), which is called a postfix representation.

    Instructions of this sort would normally be passed to an evaluator function which would convert the instructions into the final value (an integer). For instance, Python's internal bytecode includes instructions that are very similar to these.

    However, we aren't going to worry about evaluating instructions. Instead, we are just going to use our stack datatypes to compile expressions (of type expr) into a list of instructions (of type inst).1 We also want to be able to use either the functional stack implementation or the imperative stack implementation in our code, and we want to make it easy to switch between them (in other words, we don't want to hard-code the implementation choice in the compiler code).

    In OCaml, we do this kind of thing using functors. A functor is basically a kind of higher-level function on modules: you give it an input module, and it returns an output module. Often the input module represents a particular implementation of a data structure, while the output module represents one or more functions that use that implementation. That will be the case here. The input module will be a stack implementation module (either the functional Fstack module or the imperative Istack module). The output will be the expression-to-postfix compiler module that uses the particular stack module that was input to perform its computations.

    Here is the template for the functor code:

    module MakeCompiler (S : STACK) =
  struct
    (* Convert the stack to a list. *)
    let to_list (s : 'a S.t) : 'a list =
      failwith "TODO"

    (* Compile an expression to a list of postfix instructions. *)
    let compile (e : expr) : inst list =
      let rec iter in_s out_s =
        match S.pop in_s with
          | None -> to_list out_s
          | Some (i, in_s') ->
              failwith "TODO"
      in
        let input_stack = S.push (`Expr e) (S.make ()) in
        let output_stack = S.make () in
          iter input_stack output_stack
  end

    We'll discuss the implemenations of the to_list and compile functions below. Before we get to that, let's see how the functor can be used.

    The first line of the functor:

    module MakeCompiler (S : STACK) = ...

    indicates that this is a "function" from an input module called S (which must conform to the interface of the module type STACK) to an output module which uses the stack implementation in the functions it defines. Because the functor code doesn't know anything about the internal implementation of the stack module, it can only use the stack module functions make, push, and pop. Since the name given to the STACK implementation is S inside the functor, these functions are called by writing S.make, S.push, and S.pop.

    Note

    We've also put type annotations on the to_list and compile functions; we could have left them out, but type annotations are good documentation, so we'll keep them.

    This functor code can't be used without "instantiating" the functor. That means supplying a concrete module as the argument to the MakeCompiler functor.2 We can do this as follows:

    module Fcompiler = MakeCompiler (Fstack)
module Icompiler = MakeCompiler (Istack)

    This uses the MakeCompiler functor to create two modules:

    • the Fcompiler module uses the functional stack module for all of its stack operations;

    • the Icompiler module uses the imperative stack module for all of its stack operations.

    If both of the functional and imperative stack modules behave similarly, both the Fcompiler and Icompiler modules should compute the exact same outputs given the same inputs to the compile function. (Of course, the asymptotic efficiency may not be the same, though in this particular case, it is.)

    Here is a trivial test of the compiler(s):

    # let e1 = Pow (Add (Mul (Int 10, Var "x"), Var "y"), 10) ;;
val e1 : expr = Pow (Add (Mul (Int 10, Var "x"), Var "y"), 10)

# let e2 = Fcompiler.compile e1 ;;
val e2 : inst list =
  [Push_int 10; Push_var "y"; Push_var "x"; Push_int 10; Op_mul; Op_add;
   Op_pow]

# let e2 = Icompiler.compile e1 ;;
val e2 : inst list =
  [Push_int 10; Push_var "y"; Push_var "x"; Push_int 10; Op_mul; Op_add;
   Op_pow]

    Both the Fcompiler and Icompiler modules' compiler functions give the same result with the same input, as expected.

    With all of that out of the way, we can get to the two functions you need to write.

    3. The to_list function

    [10 points]

    We are going to be accumulating the postfix instructions on an output stack. In order to actually generate the list of instructions as output, we have to convert the contents of the stack to a list.

    Note

    This could have also been a function in the STACK interface. Doing it here means that we only need to define it once. The price we pay is that we have to define it in terms of the STACK interface functions; we can't directly access the internal representation of the stack (which would be faster). Programming is all about trade-offs!

    The to_list function will use the S.pop stack function to generate the output list. The topmost item in the stack will be the item at the end of the output list. Here's an example:

    # let (s : int Fstack.t) = Fstack.make ();;
val s : int Fstack.t = <abstr>

# let s1 = Fstack.push 1 s;;
val s1 : int Fstack.t = <abstr>

# let s2 = Fstack.push 2 s1;;
val s2 : int Fstack.t = <abstr>

# let s3 = Fstack.push 3 s2;;
val s3 : int Fstack.t = <abstr>

# Fcompiler.to_list s3 ;;
- : int list = [1; 2; 3]

    This would work the same way with Istack and Icompiler.

    This function is quite short, and should be done iteratively.

    4. The compile function

    [20 points]

    Now we get to the actual compiler function.

    Terminology

    We are not using the word "compiler" here to mean "something that converts source code all the way to assembly language". Instead, we mean "something that converts some representation of code to a different representation of code". In this case, the representations are the expr and inst types, and the compiler converts expr expressions into lists of insts.

    Here is the template code given above:

    let rec compile (e : expr) : inst list =
  let rec iter in_s out_s =
    match S.pop in_s with
      | None -> to_list out_s
      | Some (i, in_s') ->
          failwith "TODO"
  in
    let input_stack = S.push (`Expr e) (S.make ()) in
    let output_stack = S.make () in
      iter input_stack output_stack

    The compilation algorithm requires two stacks: an input stack and an output stack. Both will use the S module, so either both will be functional stacks or both will be imperative stacks.

    To create the input stack, an empty stack is made (using S.make ()), and the input expression is pushed onto it, wrapped in the polymorphic variant tag `Expr.3 We use polymorphic variants because the input stack will contain a mixture of expressions (which use the `Expr tag) operators (which use the `Add, `Mul, and `Pow tags), and integers (which use the `Int tag). None of these tags are used in the outputs of the function; they are only there temporarily to allow us to put different kinds of things on the input stack.

    The output stack starts off as an empty stack created by S.make (). We will add instructions to it in the course of compilation.

    The iter function works as follows:

    • If the input stack is empty, the output stack is converted to a list, and the list is returned.

    • Otherwise, it takes values off of the input stack and either:

      • pushes values back onto the input stack, or
      • pushes instructions onto the output stack.

    • It repeats until there are no more instructions on the input stack.

    There are five kinds of expressions (all wrapped in `Expr constructors on the input stack), corresponding to the five expr constructors. They are compiled as follows:

    • The Int and Var constructors can be immediately converted to Push_int and Push_var instructions and pushed onto the output stack.

    • The Add, Mul, and Pow constructors each have two subexpressions. In each case, they are compiled by taking them apart and putting their components back onto the input stack, with the operator first, then the second expression, then the first expression. The operators are represented by the polymorphic variant tags `Add, `Mul, and `Pow respectively. The expressions are wrapped in `Expr constructors. One exception is the exponent of the Pow constructor; it's not an expression but an integer. Push it back to the input stack using the `Int tag.

    To handle the `Add, `Mul, `Pow values on the input stack, push them onto the output stack as Op_add, Op_mul, and Op_pow instructions, respectively. To handle an `Int <int> value on the input stack (generated by compiling a Pow expression), convert it to a Push_int <int> instruction and push it to the output stack.

    That is the entire compiler! The only thing to add is that in some cases, you have to push multiple values back onto the input stack. This can be tedious to write:

    (* Example of pushing multiple values onto the input stack. *)

let s1 = S.push <value1> input_stack in
let s2 = S.push <value2> s1 in
  S.push <value3> s2

    Even though this code works perfectly well, there are nicer ways to write it. Note that the output of each line is the input to the next. So we can write it like this:

    S.push <value3>
  (S.push <value2>
    (S.push <value1> input_stack))

    This is better, but it's still kind of gross. We can use the |> (pipe) operator to make it look a lot nicer! The pipe operator (which is built-in to OCaml) can be defined like this:

    let ( |> ) x f = f x

    So it's just reverse function application. Big deal! Why does this matter? It's because using this operator, we can rewrite our code like this:

    (S.push <value1> input_stack
  |> S.push <value2>)
  |> S.push <value3>

    Because of the left-associativity of the |> operator, we can drop the parentheses:

    S.push <value1> input_stack
  |> S.push <value2>
  |> S.push <value3>

    We read this as "push <value1> onto the input stack, then push <value2> onto the resulting stack, then push <value3> onto the resulting stack. OCaml programmers prize concise and readable code, of which this is a good example.

    1. In fact, real implementations of programming languages often use stacks at multiple levels:

      • Parsing the concrete syntax of the language to expressions is done using a parsing stack.
      • Converting the expressions to instructions (as we are doing here) is done using a compiler stack.
      • Interpreting the list of instructions is done using a value stack.
      • Storing partially-interpreted expressions is done using a runtime stack.

      If you are interested in learning more about interpreters, take CS 131 (which also uses OCaml as the implementation language). ↩

    2. Actual functor names often start with Make (or are just the word Make) to emphasize that this is not a module but something that "makes" a module. ↩

    3. This function is an excellent example of how polymorphic variants are actually used in practice. Notice that we didn't have to define a special datatype to use the `Expr polymorphic variant constructor. ↩
